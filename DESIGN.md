# 極上UIゲームエンジン 設計ドキュメント

## プロジェクト概要

**38 Game** は、noteシリーズ向けに開発された **「1日1ゲーム」モノレポプロジェクト** です。各ゲームはHTML1枚で完結し、GitHub Pages でそのまま公開できます。

**コンセプト**: 「どうぶつの森」のような温かみのあるビジュアルと、文字が1文字ずつ心地よく表示される「極上UI体験」を、最小限のコードで実現すること。

**対象読者**: noteの技術記事を読む初心者 - 中級者のWeb開発者。コードを読んで学び、自分でカスタマイズできることを目指しています。

---

## ファイル構成

```
38_Game/
  index.html                              ... ゲームポータル（ゲーム一覧ページ）
  DESIGN.md                               ... 本設計ドキュメント
  juice/
    juice.js                              ... 共通Juiceライブラリ（アニメーション・エフェクト）
  games/
    day01-text-adventure/
      index.html                          ... Day 01: 潮風の島（テキストアドベンチャー）
  articles/
    day00-concept.md                      ... コンセプト記事
    day01-textengine.md                   ... Day 01 テキストエンジン解説記事
```

### モノレポ設計方針

- **ポータル（ルート `index.html`）**: 全ゲームへのリンクカードを一覧表示。Day が増えるたびにカードを追加する
- **各ゲーム（`games/dayXX-*/index.html`）**: HTML1枚完結。各ゲームは独立して動作し、外部依存なし
- **共通ライブラリ（`juice/juice.js`）**: 複数ゲームで使い回すアニメーション・エフェクトのユーティリティ（今後実装）
- **記事（`articles/`）**: note投稿用の原稿を管理

この構成により:
- ゲームごとにディレクトリが分かれ、Day が増えても見通しが良い
- 各ゲームは独立しているため、1つのゲームの変更が他に影響しない
- GitHub Pages へのデプロイが `git push` のみで完了する
- ポータルページから全ゲームにアクセスできる

### ポータルページ（ルート `index.html`）

- 各カード（`.game-card`）に Spring風ホバーアニメーション（`cubic-bezier(.34, 1.56, .64, 1)`）を適用

- 未公開ゲームは破線ボーダーの "coming soon" プレースホルダーで表示
- ゲームエンジンと同じ背景グラデーションアニメーション（`bgShift`）で統一感を演出

### Day 01: テキストアドベンチャーエンジン（3層構造）

```
┌─────────────────────────────────────────────┐

### データフロー（Day 01）

```

## テキストエンジン実装のポイント

### 「心地よさ」の秘密

テキスト表示の気持ちよさは **タイピング速度の緩急** によって生まれています。

```javascript
const BASE_DELAY   = 45;   // 1文字あたりの基本ウェイト（ms）
const PUNCT_DELAYS = {
  "。": 90,  // 句点で長めの間
  "、": 70,  // 読点でやや間
  "！": 90,  // 感嘆符で間
  "？": 90,  // 疑問符で間
  "…": 60,  // 三点リーダーで短い間
  "―": 30   // ダッシュでほんの少し間
};
```

- **BASE_DELAY = 45ms**: 人間が快適に読める速度。速すぎず遅すぎない絶妙な値
- **句読点の追加ウェイト**: 「。」で +90ms の間を置くことで、文の終わりに自然な「間」が生まれる
- **改行は `<br>` に変換**: `white-space: pre-wrap` と組み合わせて、シナリオデータ内の `\n` をそのまま改行として表現

### スキップ機能

メッセージウィンドウをクリック/タップすると `skipTyping()` が発動し、テキスト全文を即座に表示します。`clearTimeout(typingId)` で進行中のタイマーを停止し、全文を一括レンダリングする仕組みです。

### カーソルアニメーション

テキスト表示完了後、末尾に緑色の点滅カーソル（`<span class="cursor">`）を追加。`@keyframes blink` で `step-end` を使った離散的な点滅を実現しています。

## アニメーション設計

### 1. 背景グラデーションアニメーション (`bgShift`)

```css
background: linear-gradient(135deg, #e8f5e9 0%, #e3f2fd 50%, #fff8e1 100%);
background-size: 400% 400%;
animation: bgShift 20s ease infinite;
```

緑・青・黄の3色グラデーションが 20秒周期 でゆっくりシフトし、画面に生命感を与えます。

### 2. Spring風ホバーアニメーション

```css
transition: transform .35s cubic-bezier(.34, 1.56, .64, 1);
```

`cubic-bezier(.34, 1.56, .64, 1)` が核心です。第2制御点の y値 `1.56` が 1.0 を超えているため、アニメーション途中で目標値を **オーバーシュート** してから戻る「バネ的な動き」が生まれます。

ホバー時に `scale(1.05)` へ遷移するとき、一瞬 1.05 を超えてから着地する = **ポヨンとした弾力感** になります。

### 3. ポヨヨン決定アニメーション (`popBounce`)

```css
@keyframes popBounce {
  0%   { transform: scale(1)    }
  20%  { transform: scale(1.2)  }   /* 大きく膨らむ */
  45%  { transform: scale(.9)   }   /* 縮む */
  70%  { transform: scale(1.05) }   /* 少し膨らむ */
  100% { transform: scale(1)    }   /* 元に戻る */
}
```

選択肢を押した瞬間に再生されます。拡大 → 縮小 → 微拡大 → 元に戻る の4段階で、物理的な「押した感」を演出。duration は `0.5s` でキビキビとした応答性を保っています。

### 4. 選択肢フェードイン (`choiceFadeIn`)

```css
.choice-btn:nth-child(1) { animation-delay: .05s }
.choice-btn:nth-child(2) { animation-delay: .15s }
.choice-btn:nth-child(3) { animation-delay: .25s }
```

各ボタンが 0.1秒ずつずれて下から浮かび上がります。`translateY(8px)` からの移動と `scale(.97)` からの拡大を組み合わせ、軽やかな登場感を実現。

### 5. シーン遷移 (`#overlay`)

白い半透明オーバーレイ（`rgba(255,255,255,.95)`）を `opacity: 0 → 1 → 0` で切り替えることで、柔らかなフェードイン/アウトを実現。遷移時間は fade-out 420ms + fade-in 350ms で、テンポを損なわない速さです。

## シナリオJSON仕様

### スキーマ定義

```javascript
const scenario = {
  "シーンID": {
    text: "表示するテキスト（\\n で改行）",
    choices: [
      { label: "選択肢のラベル", next: "遷移先シーンID" },
      // 最大3個（CSSアニメーション定義上）
    ]
  }
};
```

### フィールド説明

| フィールド | 型 | 説明 |
|---|---|---|
| `text` | `string` | シーンで表示するテキスト。`\n` で改行 |
| `choices` | `Array<{label, next}>` | 選択肢の配列。空配列 `[]` でエンディング（選択肢なし） |
| `choices[].label` | `string` | ボタンに表示するテキスト |
| `choices[].next` | `string` | クリック時の遷移先シーンID |

### サンプルシナリオ構造（現在の実装）

```
start ─┬─ "起き上がる" → stand ─┬─ "海を見る"   → sea    (END)
       │                         └─ "森に向かう" → forest (END)
       └─ "もう少し寝る" → sleep ─── "起き上がる" → stand
```

5シーン構成: `start`, `stand`, `sleep`, `sea`, `forest`

### 拡張方法

**シーンを追加する場合**:
1. `scenario` オブジェクトに新しいキーを追加
2. 既存シーンの `choices[].next` を新シーンIDに変更

**選択肢を4個以上にする場合**:
- `.choice-btn:nth-child(4)` 以降の `animation-delay` をCSSに追加

**演出を追加する場合**（将来の拡張案）:
- `text` の代わりにオブジェクト `{ text, bg, bgm, speaker }` にすればシーンごとの背景変更やBGM指定が可能

## note記事用 解説ポイント（初心者向け）

### 1. たった45msの魔法 -- テキスト表示が「気持ちいい」理由

`setTimeout` の再帰呼び出しで1文字ずつ表示するテクニックは基本的ですが、句読点ごとにウェイトを変える工夫が「読んでいて心地いい」体験を生みます。BASE_DELAY の 45ms は試行錯誤で見つけた値で、これより速いと「流れていく」感じになり、遅いと「待たされる」感じになります。この「ちょうどいい」を見つけるプロセス自体がUI設計の醍醐味です。

### 2. CSS cubic-bezier でバネアニメーション -- JSなしで「ポヨン」を作る

物理シミュレーションライブラリを使わなくても、`cubic-bezier(.34, 1.56, .64, 1)` という4つの数字だけで「バネっぽい動き」が作れます。ポイントは第2制御点の y値を 1.0 より大きくすること。これだけでオーバーシュート（行き過ぎてから戻る）が起き、生き物のような弾力感が生まれます。

### 3. HTML1枚で完結するゲームエンジンという設計思想

フレームワークもビルドツールも不要。HTMLファイル1つをブラウザにドラッグ&ドロップするだけで動く。この「ゼロ依存」設計は、初心者にとって最大の学習メリットです。コードの全体像が1ファイルで見渡せるため、「何がどこで起きているか」が迷子にならない。GitHub Pages にプッシュすればすぐ公開できるのも大きな利点です。モノレポ構成にすることで、Day が増えても各ゲームは独立して管理でき、ポータルページから全ゲームにワンクリックでアクセスできます。

## 今後の拡張アイデア

- **BGM / SE**: `<audio>` タグとシーンデータへの `bgm` / `se` フィールド追加で、場面ごとの音楽・効果音を再生
- **立ち絵 / 背景画像**: シーンデータに `bg` / `character` フィールドを追加し、CSS で切り替え
- **セーブ / ロード**: `localStorage` に現在のシーンIDを保存し、再訪問時に復帰
- **テキスト演出の強化**: 文字色変更、シェイク、フェードなどの inline 演出タグ対応
- **分岐フラグ**: グローバルな状態変数を持ち、過去の選択に応じてテキストや選択肢を変化させる
- **外部JSONシナリオ読み込み**: `fetch()` でシナリオを外部ファイルから読み込み、エンジンとデータを分離
- **タッチ操作最適化**: スワイプでのテキスト送り、長押しでオートモードなどモバイル向けUX強化
